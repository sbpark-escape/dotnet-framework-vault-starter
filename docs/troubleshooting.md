# Troubleshooting

Use this document when Vault lookup fails after configuration is added, especially under IIS or Windows Server. It focuses on safe diagnostics that help identify missing configuration, permission problems, path mistakes, and mount point issues without printing secret values.

Start with safe diagnostics:

```csharp
System.Diagnostics.Trace.TraceInformation(VaultDiagnostics.CreateReport());
```

The output reports status only. It does not print secret values.

## Environment Variables Are Not Visible in IIS

Symptoms:

- diagnostics show `VAULT_ADDR=MISSING` or `VAULT_AUTH_METHOD=MISSING`
- the same variables are visible in an administrator PowerShell window
- the app works from a console but fails under IIS

Check:

- confirm whether the variable was set at Process, User, or Machine scope
- recycle the exact application pool that hosts the app
- avoid relying on an interactive administrator user's environment
- confirm the app pool identity and whether User-scope variables belong to that identity
- use a protected diagnostics endpoint temporarily to compare IIS process status

## setx Was Used but IIS Still Sees Old Values

Symptoms:

- `setx VAULT_ADDR ...` completed successfully
- a new command prompt sees the value
- the already-running IIS worker process still reports `MISSING` or an old value

Check:

- `setx` changes future processes, not already-running processes
- recycle the app pool after using `setx`
- restart the Windows Service if the app is not hosted in IIS
- schedule an IIS restart only if app pool recycle is not enough in your environment

## App Pool Identity vs Machine and User Variables

Symptoms:

- Machine-scope variables work, but User-scope variables do not
- variables work under one app pool and fail under another

Check:

- Machine-scope variables are usually simplest for IIS, with appropriate server access controls
- User-scope variables must be set for the actual app pool identity, not the administrator account
- custom service accounts and built-in identities can have different profiles and environment inheritance behavior
- deployment tools may overwrite the application pool environment block

## VAULT_ROLE_ID Is SET but VAULT_SECRET_ID Is EMPTY

Symptoms:

- diagnostics show `VAULT_ROLE_ID=SET`
- diagnostics show `VAULT_SECRET_ID=EMPTY`
- provider creation fails for AppRole auth

Check:

- an empty string is different from a missing variable, but both are invalid for AppRole
- confirm the deployment step that writes the secret id did not receive an empty value
- check quoting in PowerShell or deployment templates
- rotate the `secret_id` if there is any chance the value was exposed

## Vault Address Missing

Symptoms:

- configuration warning for missing Vault address
- provider creation fails before calling Vault

Check:

- `VAULT_ADDR`
- `Vault:Address`
- app pool recycle after setting machine variables
- do not include a real internal Vault URL in public issues or logs

## Permission Denied

Symptoms:

- Vault returns HTTP 403
- error text includes `permission denied`

Check:

- Vault policy path uses `secret/data/...` for KV v2 policies
- application code path omits the mount prefix, for example `legacy-app/database`
- AppRole has the expected policy
- token generated by AppRole is not expired
- the AppRole and the policy are configured in the expected Vault namespace or environment
- the application is using the expected auth method and not a local development token

## Path Not Found

Symptoms:

- Vault returns HTTP 404
- error text mentions missing value or not found

Check:

- the secret exists
- the mount point is correct
- the code path matches the stored secret path
- KV engine version is v2
- the key exists inside the secret data object, for example `connectionString`

## Mount Point Errors

Symptoms:

- reads work in Vault CLI but not in the app
- errors mention unsupported or invalid path

Check:

- `VAULT_MOUNT_POINT` should be the mount name only, for example `secret`
- `GetSecret("legacy-app/database", "connectionString")` should not include `secret/data`
- Vault policy paths for KV v2 usually include `data`, but application lookup paths do not

## KV v1 and KV v2 Path Confusion

Symptoms:

- Vault CLI examples appear to work, but application reads fail
- policies use one path shape and application code uses another
- errors mention not found, invalid path, or unsupported path

Check:

- this starter currently assumes KV v2
- `VAULT_MOUNT_POINT` is only the mount name, for example `secret`
- application code should call `GetSecret("legacy-app/database", "connectionString")`
- KV v2 Vault policies often use `secret/data/legacy-app/*`
- do not pass `secret/data/legacy-app/database` as the application path

## Works Locally, Fails Under IIS

Check:

- app pool identity
- machine-level environment variables
- app pool recycle
- TLS root certificate trust under the server account
- proxy/firewall rules from the server
- whether the deployment copied the expected `Web.config`

## TLS or Certificate Errors

Symptoms:

- network or TLS exception before Vault returns a normal HTTP status
- certificate chain or trust errors on Windows Server
- works on a developer machine but not on the server

Check:

- the server trusts the certificate authority used by Vault
- the certificate hostname matches `VAULT_ADDR`
- the service account or app pool identity can access the relevant Windows certificate stores
- outbound proxy or firewall rules allow access to Vault
- do not bypass certificate validation in application code as a workaround

## Secret Values in Logs

Do not log raw values returned from Vault.

Use:

```csharp
Trace.TraceInformation("connectionString: " + VaultDiagnostics.Status(connectionString));
```

Avoid:

Do not pass the secret variable itself to `Trace`, `Console`, exception messages, HTTP responses, or application logs.

## Before Opening a Public Issue

Include:

- .NET Framework version
- application type, for example WebForms, ASMX, Windows Service, or console
- auth method, either Token or AppRole
- diagnostics output with only `SET`, `EMPTY`, `MISSING`, or `MASKED`

Do not include:

- real Vault addresses
- real tokens
- real role ids or secret ids
- real connection strings
- internal domains or hostnames
